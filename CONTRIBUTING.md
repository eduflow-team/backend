# 팀 프로젝트 깃 컨벤션 가이드라인

효율적인 협업과 신속한 개발을 위한 표준 Git 운영 프로세스입니다. 모든 팀원은 아래 규칙을 준수합니다.

## 1. 브랜치 전략 (Branch Strategy)

브랜치는 상시 운영 브랜치(2개)와 작업 브랜치(2개)로 분리하여 운영합니다. 모든 작업 브랜치는 `develop` 브랜치에서 분기합니다.

### 📌 상시 운영 브랜치

- `main` : 프로덕션 배포 가능한 최신의 안정적인 코드만 관리 (직접 Push 절대 금지)
- `develop` : 다음 버전을 위한 개발 코드가 모이는 기준 브랜치 (테스트 및 통합 환경)

### 📌 작업 브랜치 (명명 규칙 통일)

브랜치명은 소문자로 시작하며, 슬래시(`/`) 뒤에 `간단한설명`을 붙여 생성합니다. 터미널 환경 오류 방지를 위해 `#` 기호는 사용하지 않습니다.

- `feat/기능명` : 새로운 기능 개발 (예: `feat/login-api`, `feat/user-schema`)
- `fix/버그명` : 버그 수정 (예: `fix/null-pointer`, `fix/cors-error`)

## 2. 이슈(Issue) & 풀 리퀘스트(PR) 컨벤션

초기 빌드업 속도를 위해 **이슈 발행은 필수 사항이 아닙니다.** 큰 단위의 태스크 트래킹이나 기록이 필요한 경우에만 선택적으로 생성합니다.

### 📝 이슈 제목 규칙 (생성 시에만 적용)

```
[타입] 작업 내용
```

- `[Feat] 카카오 로그인 기능 구현`
- `[Chore] 패키지 의존성 라이브러리 추가`

### 🔀 Pull Request 제목 규칙

```
[타입] 작업 내용
```

- `[Feat] 로그인 API 엔드포인트 구현`
- `[Fix] 로그인 시 Null Pointer 에러 수정`

> 💡 **이슈를 생성했을 경우의 팁**
> PR 본문에 `Closes #이슈번호`를 작성하면, 해당 PR이 머지될 때 연결된 Issue가 자동으로 종료됩니다.

## 3. 커밋 메시지 컨벤션 (Conventional Commits)

글로벌 표준인 `type: subject` 형태로 통일합니다. 커밋 메시지만 보고도 변경 사항을 파악할 수 있도록 작성합니다.

### ✉️ 커밋 메시지 기본 구조

```
type: 작업 내용
```

> ⚠️ 주의: `type:` 뒤에는 반드시 **한 칸 공백(space)**을 둡니다. 영문일 경우 **소문자**로 작성합니다.

| 타입 | 설명 | 예시 |
|---|---|---|
| `feat` | 새로운 기능 추가 | `feat: 카카오 로그인 API 연동` |
| `fix` | 버그 수정 | `fix: 로그인 시 널 포인터 에러 수정` |
| `docs` | 문서 수정 (README, 위키 등) | `docs: API 명세서 리드미 업데이트` |
| `style` | 코드 포맷팅, 세미콜론 누락 등 (기능 변경 X) | `style: 코드 인덴트 및 포맷 정렬` |
| `refactor` | 코드 리팩토링 (가독성/구조 개선) | `refactor: 로그인 검증 로직 모듈화` |
| `test` | 테스트 코드 추가 및 수정 | `test: 인증 컨트롤러 유닛 테스트 추가` |
| `chore` | 빌드 설정, 패키지 매니저 변경, 단순 관리 등 | `chore: fastapi 패키지 의존성 추가` |

## 4. 핵심 협업 및 코드 리뷰 수칙

- **Repository 보호 설정 (Branch Protection)** : `main` 및 `develop` 브랜치는 직접 Push가 불가능하도록 설정하며, 반드시 PR을 통해서만 코드가 반영되도록 강제합니다.
- **PR 생성 및 리뷰어 지정** : PR 생성 시 본인을 제외한 팀원들을 리뷰어로 지정합니다. 요청 전, 'Files changed' 탭에서 셀프 리뷰를 진행하여 불필요한 주석이나 디버깅용 코드(`print` 등)를 제거합니다.
- **리뷰 및 머지 조건** : 최소 1명 이상의 팀원에게 Approve(승인)를 받아야 머지가 가능합니다. 변경 사항이 200~400줄 이내가 되도록 자주 쪼개어 PR을 생성하는 것을 권장합니다.
- **머지 방식 (Merge Strategy)** : 커밋 로그를 깔끔하게 유지하기 위해 자잘한 작업 커밋들은 하나로 묶어 머지하는 **Squash and merge** 방식을 기본으로 사용합니다.

### 🧹 로컬 브랜치 수시 정리

머지가 완료된 브랜치는 아래 명령으로 로컬을 주기적으로 정리합니다.

```bash
git checkout develop
git pull origin develop
git branch -d feat/login-api
git fetch --prune
```

저장소 내 [`.github/pull_request_template.md`](.github/pull_request_template.md) 파일로 등록하여 PR 생성 시 자동으로 사용되도록 구성합니다.

```markdown
## 📌 개요
-

## 📝 주요 변경 사항
-

## 📸 스크린샷 및 테스트 결과 (선택)
-

## 🔗 관련 이슈 (선택)
- Closes #
```

## 6. 충돌(Conflict) 발생 시 해결 프로세스

작업 도중 `develop` 브랜치의 변경 사항으로 인해 충돌이 발생하면, 원격이 아닌 **본인의 로컬 작업 브랜치**에서 충돌을 해결한 뒤 다시 push합니다.

1. 본인의 로컬 작업 브랜치로 이동 및 최신 `develop` 코드 가져오기
   ```bash
   git checkout feat/login-api
   git pull origin develop
   ```
2. 에러가 발생한 파일의 충돌 코드(`<<<<<<< HEAD`와 `>>>>>>>`)를 직접 확인 후 수정
3. 수정 완료 후 다시 스테이징 및 커밋, 푸시 진행
   ```bash
   git add .
   git commit -m "chore: develop 브랜치 머지 충돌 해결"
   git push origin feat/login-api
   ```

## 7. 코드 리뷰 피드백 규칙 (P-태그)

리뷰어의 의견이 필수 수정 사항인지, 단순 의견 제시인지 명확히 구분하여 소통 비용을 최소화합니다.

### P1 (Must Amend)
기능적 오류, 버그 가능성이 높은 심각한 코드. **무조건 수정 후 재리뷰**를 받아야 머지 가능.
> 예시: "P1: 이 부분 에러 처리가 누락되어 null 값이 들어오면 서버가 터질 수 있습니다. 예외 처리 추가해주세요."

### P2 (Should Amend)
구조적 개선, 가독성, 컨벤션 준수 등을 위한 적극 권장 사항. 반영을 권장하되 불가피한 경우 사유 작성 후 머지 가능.
> 예시: "P2: 중복되는 검증 로직인데 utils 폴더로 분리하는 게 어떨까요?"

### P3 (Comment)
단순 질문, 사소한 팁, 칭찬 등. 반영 여부는 작업자가 자율적으로 판단하며 피드백 없이 머지 가능.
> 예시: "P3: 가독성이 깔끔하고 좋네요! 혹시 이 라이브러리를 사용하신 특별한 이유가 있나요?"
```
